# AR Chess: IoT Система управления шахматными стендами

AR Chess — это аппаратно-программный комплекс для игры в шахматы на физических досках с поддержкой дополненной реальности и удаленного взаимодействия. Система объединяет физические шахматные стенды, серверную логику и веб-интерфейс для мониторинга игр.

## Архитектура системы

Система построена на событийно-ориентированной микросервисной архитектуре. Связующим звеном между всеми компонентами является MQTT-брокер.

- **Backend (Lobby Service)**: Управляет пользователями, базой данных и жизненным циклом игровых сессий.
- **Game Engine**: Изолированный процесс, отвечающий за шахматную логику (валидация ходов, детекция мата, обработка рокировок).
- **Frontend**: Веб-интерфейс для игроков и зрителей, работающий через WebSockets.
- **Stand Agents / Mocks**: Устройства (или их программные симуляторы), передающие состояние физической доски.

## Описание компонентов

### Lobby Service (`lobby_service`)
- Реализован на **FastAPI**.
- Использует асинхронный драйвер **asyncpg** для работы с PostgreSQL.
- **Game Orchestrator**: Модуль внутри лобби, который через `subprocess` запускает экземпляры игровых движков и моков.
- **MQTT Listener**: Слушает топик `lobby/game_over` для автоматической очистки ресурсов по завершении игры.

### Game Engine (`game_engine_chess`)
- Основан на библиотеке **python-chess**.
- Реализует логику **DESYNC** (рассинхрон): если состояние физической доски не совпадает с виртуальной, движок подсвечивает ошибки.
- Обрабатывает ситуацию «взятия» фигуры, когда два маркера кратковременно оказываются на одной клетке.

### Frontend (`frontend`)
- Написан на **React** с использованием **Tailwind CSS**.
- **Chessground**: Интеграция высокопроизводительной доски (движок Lichess) для плавной визуализации ходов.
- Подключается к MQTT через WebSockets для получения мгновенных обновлений состояния.

## Установка и запуск

### Предварительные требования
- Docker
- Docker Compose

### Развертывание
1. Склонируйте репозиторий.
2. В корневой директории выполните сборку и запуск:
```bash
docker-compose up --build -d
```
3. После запуска сервисы будут доступны по следующим адресам:
   - **Frontend**: `http://localhost`
   - **Backend API (Swagger)**: `http://localhost:8000/docs`
   - **MQTT Broker**: `localhost:1883` (TCP), `localhost:9001` (WebSockets)

## MQTT Контракт

Для взаимодействия компонентов используются следующие топики:

### 1. Входные данные от стендов
**Топик**: `game/{game_id}/{stand_id}/markers`
**Формат JSON**:
```json
{
  "matrix": [
    {"id": 6, "cords": [0, 4]},
    {"id": 11, "cords": [6, 4]}
  ]
}
```
*Где `id` 1-6 — белые фигуры, 11-16 — черные.*

### 2. Состояние игры (для Frontend)
**Топик**: `game/{game_id}/state`
**Формат JSON**:
```json
{
  "fen": "rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR",
  "engine_state": "PLAYING",
  "is_white_turn": true,
  "status": [
    {"square": [0, 4], "description": "check"}
  ]
}
```

## Переход от симуляции к реальным стендам

Текущая версия системы использует `stand_mock_service` для демонстрации. Чтобы интегрировать реальное оборудование, выполните следующие шаги:

### 1. Модификация Orchestrator
В файле `lobby_service/app/services/game_orchestrator.py` необходимо отключить автоматический запуск подпроцессов моков. Вместо `subprocess.Popen` для мока, оркестратор должен отправить управляющую команду в MQTT:
```python
# Вместо запуска процесса мока:
lobby_mqtt.client.publish(f"stands/{stand_id}/commands", json.dumps({
    "action": "start_game",
    "game_id": game_id,
    "role": role
}))
```

### 2. Подготовка Hardware Agent
На физическом устройстве (Raspberry Pi / Orange Pi) должен быть запущен агент, который:
1. Подписан на топик управления `stands/{hardware_id}/commands`.
2. При получении команды `start_game` начинает цикл обработки видеопотока с камеры.
3. Формирует JSON с координатами распознанных маркеров и публикует его в топик `game/{game_id}/{stand_id}/markers`.

### 3. Регистрация устройств
- Реальное устройство должно использовать свой уникальный Hardware ID (например, серийный номер).
- При включении устройство генерирует ПИН-код и отправляет его на `/api/v1/stands/register-pin`.
- Пользователь привязывает стенд в Dashboard, после чего `stand_id` в системе будет соответствовать реальному «железу».

### 4. Игровой движок
Компонент `game_engine_chess` **не требует изменений**. Он универсален и обрабатывает любые данные, поступающие в соответствии с форматом MQTT-контракта, независимо от того, получены они от симулятора или с камеры реального стенда.

## Безопасность и эксплуатация

- **Авторизация**: Реализована на базе JWT (JSON Web Tokens). Токены хранятся в `localStorage` браузера.
- **База данных**: Миграции управляются через **Alembic**. При изменении моделей данных всегда создавайте новую миграцию.
- **MQTT**: Брокер Mosquitto настроен на одновременную работу по протоколу MQTT (для Python-сервисов) и WebSockets (для браузерного фронтенда).
- **Очистка сессий**: Игровой движок завершает себя автоматически при фиксации мата или пата, посылая сигнал в лобби для освобождения ресурсов стендов.
