# AR Chess IoT: Протокол и документация системы управления шахматными стендами

AR Chess представляет собой распределенную программно-аппаратную платформу для организации шахматных партий на физических досках с использованием компьютерного зрения и дополненной реальности (AR). Система позволяет двум игрокам взаимодействовать удаленно, используя физические фигуры своего цвета, в то время как фигуры оппонента отображаются с помощью проекционных технологий.

## Архитектурные принципы

Система спроектирована на основе событийно-ориентированной архитектуры (Event-Driven Architecture) с центральным узлом обмена сообщениями (MQTT Broker). 

**Ключевые особенности:**
*   **Разделение ответственности:** Стенды (Edge) отвечают за детекцию и визуализацию, центральный сервер (Core) — за шахматную логику и арбитраж.
*   **Изоляция сессий:** Для каждой активной игры Lobby Service запускает отдельный изолированный процесс Game Engine, что гарантирует стабильность системы при масштабировании.
*   **Hybrid Sync:** Логика движка учитывает, что на каждом стенде присутствуют только фигуры одного цвета.

## Описание программных модулей

### Lobby Service
Центральный управляющий сервис. Выполняет функции:
*   Управление пользователями и сессиями (JWT авторизация).
*   Регистрация оборудования через процедуру Pairing (ПИН-коды).
*   Оркестрация: динамический запуск и остановка процессов `game_engine`.

### Game Engine
Автономный вычислительный модуль:
*   Подключается к конкретной игровой сессии по MQTT.
*   Валидирует ходы, используя библиотеку `python-chess`.
*   Формирует AR-слой: массив координат для подсветки возможных ходов, шахов и ошибок расстановки (DESYNC).

### Stand Agent
Клиентское ПО для Raspberry Pi:
*   **CV-пайплайн:** Захват видео, калибровка перспективы и распознавание ArUco-маркеров.
*   **AR-рендер:** Отрисовка графического интерфейса для проектора.
*   **MQTT-клиент:** Передача состояния доски только при фиксации стабильных изменений.

## Документация REST API

Базовый URL: `http://<server_ip>:8000/api/v1`
Все защищенные запросы требуют заголовок: `Authorization: Bearer <JWT_TOKEN>`

### Авторизация
*   `POST /auth/register` — Регистрация (username, password).
*   `POST /auth/login` — Вход, получение токена.
*   `GET /auth/me` — Проверка статуса авторизации.

### Стенды
*   `POST /stands/register-pin` — Регистрация ПИН-кода (вызывается стендом при включении).
*   `POST /stands/activate` — Привязка стенда к аккаунту (вызывается пользователем).
*   `GET /stands/my` — Список устройств текущего пользователя.

### Игры
*   `POST /games/invite` — Создание матча. Параметр `use_mocks: true` позволяет запустить симуляцию.
*   `GET /games/active` — Мониторинг активных сессий.
*   `POST /games/{game_id}/surrender` — Принудительная остановка игры.

## Спецификация протокола MQTT

### Формат координат
Координаты передаются в виде массива `[row, col]`, где значения лежат в диапазоне `0-7`.
*   `0,0` соответствует клетке `A1`.
*   `7,7` соответствует клетке `H8`.

### Идентификаторы фигур (Figure ID)
*   **Белые:** 1 (Пешка), 2 (Конь), 3 (Слон), 5 (Ферзь), 6 (Король), 9 (Ладья).
*   **Черные:** Тот же набор, но с префиксом 10 (например, 11 — черная пешка).

### Основные топики
1.  **Управление стендом:** `stands/{stand_id}/commands`
    *   Payload: `{"action": "start_game", "game_id": "str", "role": "white|black"}`
2.  **Передача маркеров:** `game/{game_id}/{stand_id}/markers`
    *   Payload: `{"matrix": [{"id": int, "cords": [int, int]}]}`
3.  **Состояние доски:** `game/{game_id}/state`
    *   Payload: `{"fen": "str", "engine_state": "str", "status": [{"square": [r, c], "description": "str"}]}`

## Развертывание системы

### Настройка сервера
1. Создайте файл `.env` в корне проекта:
   ```env
   POSTGRES_USER=user
   POSTGRES_PASSWORD=password
   POSTGRES_DB=ar_games
   DATABASE_URL=postgresql+asyncpg://user:password@db:5432/ar_games
   MQTT_BROKER_HOST=mqtt
   DOCKER_ENV=1
   ```
2. Запустите сборку:
   ```bash
   docker-compose up --build -d
   ```

### Настройка фронтенда (для внешнего доступа)
Перед сборкой убедитесь, что в `VITE_API_URL` указан сетевой IP-адрес сервера, а не `localhost`.

## Интеграция физического оборудования

Для корректного взаимодействия реального стенда с облачной логикой Агент должен соблюдать следующие условия:

1.  **Алгоритм стабилизации (Debounce):** Агент не должен публиковать данные в MQTT, пока расстановка на доске не будет стабильна в течение 0.5–0.8 секунд. Это предотвращает ложные срабатывания DESYNC при перекрытии камеры рукой.
2.  **Трансформация координат:**
    *   Стенд Белых (White): должен передавать координаты как есть (0-7).
    *   Стенд Черных (Black): должен инвертировать ряд (`row = 7 - row`), если камера расположена со стороны игрока.
3.  **Обработка взятия:** При получении в топике `state` статуса с описанием `capture_cleanup`, Агент должен визуализировать через проектор требование убрать физическую фигуру с указанной клетки.

## Устранение неполадок

*   **Ошибка расстановки (DESYNC) на старте:** Проверьте, что в данных от камер координаты начинаются с 0, а не с 1. Убедитесь, что ID фигур соответствуют спецификации.
*   **Задержка ходов:** Проверьте загрузку CPU на Raspberry Pi. Рекомендуемая частота отправки данных в MQTT — не более 5 Гц.
*   **Отсутствие редиректа:** Убедитесь, что порт 9001 (WebSockets) открыт в брандмауэре сервера и Mosquitto настроен на прием WS-соединений.
*   **Проблема при взятии фигур:** Проверьте логи движка. Движок отдает приоритет стенду активного игрока. Если Белые съели Черных, ход будет засчитан, даже если Черный стенд еще видит свою фигуру.
